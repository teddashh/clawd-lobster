## 1. Schema Design: The Knowledge Graph

The current flat schema is insufficient. I propose a graph-based schema that models knowledge as a network of entities and relationships. This provides the flexibility and depth required to serve as a "deep brain."

### Core Principles:
- **Graph-based:** Nodes (entities) and Edges (relationships) are first-class citizens.
- **Provenance-first:** Every piece of data is traceable to a `source`.
- **Vector-native:** Semantic search is a core, not an add-on.
- **Flexible & Extensible:** Key-value properties allow the schema to adapt without DDL changes.
- **Temporal:** Lifecycle and validity dates are built-in.

### Proposed DDL for "The Vault"

```sql
-- DDL for "The Vault" (L3 Deep Brain)

-- 1. SOURCES: The origin of every piece of knowledge
CREATE TABLE sources (
    source_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    source_type VARCHAR2(50) NOT NULL, -- e.g., 'file', 'email', 'url', 'user_input', 'api', 'legacy_import'
    source_uri VARCHAR2(2048) UNIQUE NOT NULL, -- e.g., file path, URL, message-id, oracle:knowledge:<id>
    source_hash VARCHAR2(256), -- SHA256 hash of the content for integrity and deduplication
    ingested_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    metadata CLOB -- JSON blob for extra source metadata (e.g., email headers)
);
COMMENT ON TABLE sources IS 'Stores the provenance of all data in the knowledge graph. Every piece of information must trace back to a source.';


-- 2. NODES: The entities (vertices) of the knowledge graph
CREATE TABLE nodes (
    node_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    node_type VARCHAR2(50) NOT NULL, -- e.g., 'Person', 'Project', 'Document', 'Email', 'Meeting', 'Decision', 'Topic', 'Organization'
    primary_name VARCHAR2(1024), -- A human-readable name, title, or summary
    status VARCHAR2(50) DEFAULT 'raw' NOT NULL, -- 'raw' -> 'extracted' -> 'synthesized' -> 'accepted' -> 'superseded'
    source_id NUMBER, -- The primary source this node was created from
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    modified_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    content_valid_from TIMESTAMP WITH TIME ZONE, -- The start of the time range the node's content is about
    content_valid_to TIMESTAMP WITH TIME ZONE,   -- The end of that time range
    CONSTRAINT fk_nodes_source FOREIGN KEY (source_id) REFERENCES sources(source_id)
);
CREATE INDEX idx_nodes_type_name ON nodes(node_type, primary_name);
COMMENT ON TABLE nodes IS 'Represents the core entities (vertices) of the knowledge graph.';


-- 3. CONTENT_CHUNKS: Stores text content and its vector embedding for semantic search
CREATE TABLE content_chunks (
    chunk_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    node_id NUMBER NOT NULL, -- The document/email/etc. node this chunk belongs to
    chunk_order NUMBER DEFAULT 0,
    chunk_text CLOB NOT NULL,
    embedding VECTOR(1536), -- The vector embedding for semantic search
    token_count NUMBER,
    source_id NUMBER, -- Can be a more granular source than the parent node
    CONSTRAINT fk_chunks_node FOREIGN KEY (node_id) REFERENCES nodes(node_id) ON DELETE CASCADE,
    CONSTRAINT fk_chunks_source FOREIGN KEY (source_id) REFERENCES sources(source_id)
);
CREATE INDEX idx_chunks_embedding ON content_chunks(embedding) INDEXTYPE IS VECTOR;
COMMENT ON TABLE content_chunks IS 'Stores text chunks and their vector embeddings. A node can be broken into many chunks.';


-- 4. EDGES: The relationships (edges) connecting the nodes
CREATE TABLE edges (
    edge_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    source_node_id NUMBER NOT NULL,
    target_node_id NUMBER NOT NULL,
    edge_type VARCHAR2(100) NOT NULL, -- e.g., 'discusses', 'authored_by', 'belongs_to', 'references', 'decided_on', 'mentions'
    status VARCHAR2(50) DEFAULT 'active' NOT NULL, -- 'active', 'superseded', 'inferred'
    source_id NUMBER, -- Provenance for the relationship itself (e.g., the email that established the link)
    confidence_score NUMBER(5,4) DEFAULT 1.0, -- For inferred relationships
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_edges_source_node FOREIGN KEY (source_node_id) REFERENCES nodes(node_id) ON DELETE CASCADE,
    CONSTRAINT fk_edges_target_node FOREIGN KEY (target_node_id) REFERENCES nodes(node_id) ON DELETE CASCADE,
    CONSTRAINT fk_edges_source FOREIGN KEY (source_id) REFERENCES sources(source_id)
);
CREATE INDEX idx_edges_all ON edges(source_node_id, target_node_id, edge_type);
COMMENT ON TABLE edges IS 'Represents the directed relationships (edges) between nodes in the knowledge graph.';


-- 5. NODE_PROPERTIES & EDGE_PROPERTIES (Key-Value for flexibility)
CREATE TABLE node_properties (
    property_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    node_id NUMBER NOT NULL,
    property_key VARCHAR2(256) NOT NULL,
    property_value CLOB,
    CONSTRAINT fk_nodeprops_node FOREIGN KEY (node_id) REFERENCES nodes(node_id) ON DELETE CASCADE,
    UNIQUE (node_id, property_key)
);
COMMENT ON TABLE node_properties IS 'Provides a flexible key-value store for rich, queryable metadata on nodes.';

CREATE TABLE edge_properties (
    property_id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
    edge_id NUMBER NOT NULL,
    property_key VARCHAR2(256) NOT NULL,
    property_value CLOB,
    CONSTRAINT fk_edgeprops_edge FOREIGN KEY (edge_id) REFERENCES edges(edge_id) ON DELETE CASCADE,
    UNIQUE (edge_id, property_key)
);
COMMENT ON TABLE edge_properties IS 'Provides a flexible key-value store for rich metadata on edges (e.g., roles in a meeting).';
```

## 2. `absorb` Skill Integration

The `absorb` skill must become more than an importer; it must become a graph builder. It will feed a sophisticated, multi-stage **Enrichment Pipeline**.

**Data Flow Diagram:**
`User Input (file/url)` -> `absorb` -> `1. Create Source` -> `2. Create Raw Node` -> **`Enrichment Pipeline`** -> `7. Update Node Status`

**Enrichment Pipeline Steps:**
1.  **Create Source:** A `sources` record is created with the file URI/URL and a content hash.
2.  **Create Raw Node:** A `nodes` record is created (e.g., type='Document', status='raw') linking to the source.
3.  **Chunking:** Content is split into semantically coherent `content_chunks`.
4.  **Embedding:** Each `chunk_text` is sent to an embedding model, and the resulting vector is stored in `content_chunks.embedding`.
5.  **Entity & Relation Extraction:** An LLM analyzes the content to:
    *   Identify named entities (people, projects, organizations).
    *   Identify relationships between them ('A works for B', 'C discussed D').
6.  **Graph Population:**
    *   For each extracted entity, find or create a `node`.
    *   Create `edges` linking the source document node to these entity nodes (e.g., `[Document Node] -mentions-> [Person Node]`).
    *   Create `edges` between the extracted entities themselves (e.g., `[Person Node] -works_for-> [Organization Node]`). The `source_id` for these new edges is the document they were found in.
7.  **Status Update:** The original node's status is promoted from `raw` to `extracted` or `synthesized`.

This process turns a flat document into a rich, interconnected part of the knowledge graph.

## 3. Migration Plan

Migrating the legacy `knowledge` table is a one-time, scripted process that uses the same Enrichment Pipeline.

1.  **Setup:** Deploy the new schema tables alongside the old `knowledge` table.
2.  **Migration Script:** A Python script will iterate through each row of the `knowledge` table.
    a.  **Create Source:** Create a `sources` record with `source_type='legacy_import'` and `source_uri='oracle:knowledge:<id>'`.
    b.  **Create Node:** Create a primary `node` for the item. Infer `node_type` from the old `category` column. Set `status` to `accepted` as this is existing, trusted knowledge.
    c.  **Run Enrichment:** Pass the `content` field through the same **Chunking, Embedding, and Extraction** pipeline defined for the `absorb` skill. This creates the associated `content_chunks`, entity `nodes`, and `edges`.
    d.  **Migrate Tags:** For each tag in the old `tags` column, find/create a `Topic` node and create an edge: `[Knowledge Node] -has_topic-> [Topic Node]`.
3.  **Verification:** Run validation queries to ensure no data was lost (e.g., row counts, spot checks).
4.  **Cutover:** Once verified, application logic is switched to use the new schema exclusively, and the old `knowledge` table is deprecated.

## 4. Onboarding UX

Clarity and ease-of-use are paramount for this optional feature.

**Documentation (`README.md`):**
We must clearly separate the default experience from the "power-up".

> **Core Memory (L1+L2): Simple & Portable**
> Out of the box, `clawd-lobster` uses a local SQLite database and a Git-based wiki for a fast, private, and portable memory system. This is all you need to get started.
>
> **The Vault (Optional L3): Your Deep Brain**
> For those who want to build a true "second brain," `clawd-lobster` can connect to an Oracle Autonomous Database. This optional L3 layer, "The Vault," transforms your agent from a task assistant into a knowledge archivist.
> *   **Benefits:** Unlocks massive scale, automatic relationship discovery, and the ability to answer deep, complex questions across years of data.
> *   **Prerequisites:** Requires an Oracle Cloud account (a generous "Always Free" tier is available).

**Painless Setup:**
A guided CLI tool will handle configuration.
```bash
npx clawd-lobster-setup --configure-vault
```
The tool will:
1.  Ask for the Oracle connection details (wallet, user, password).
2.  Test the connection.
3.  Detect if the schema exists and offer to run the DDL scripts to create it.
4.  Securely store the credentials and set the `L3_ENABLED=true` flag in the project's configuration.

## 5. Query Architecture

Agents must not query databases directly. They will interact with a single, intelligent **Query Orchestrator** API that abstracts the memory layers.

**Query Flow:**
1.  An agent poses a natural language question to the `QueryOrchestrator`.
    *   *Example: "What did David and I discuss about Project X in 2023?"*
2.  **Query Decomposition (LLM-powered):** The orchestrator breaks the query into its constituent parts:
    *   **Entities:** `Person:'David'`, `Project:'Project X'`
    *   **Relationships:** `discuss`
    *   **Timeframe:** `2023-01-01` to `2023-12-31`
    *   **Semantic Core:** "discussion about Project X"
3.  **Multi-Layer Query Execution:**
    *   **L3 (Graph Query):** The orchestrator first attempts a deep query on the Vault. It translates the decomposed parts into a `SQL/PGQ` graph query, combining graph traversal with vector search.
    *   **L2 (Keyword Search):** It searches the Git Wiki for canonical, curated documents matching keywords like "Project X".
    *   **L1 (Operational Query):** It queries the SQLite ledger for recent, relevant operational data (e.g., tasks or decisions about Project X created today).
4.  **Synthesis & Citation (LLM-powered):**
    *   The orchestrator gathers results from all layers.
    *   It uses an LLM to synthesize a single, coherent answer.
    *   Crucially, the answer is **cited** with links back to the original `source_uri` from each piece of evidence, providing full traceability for the user.

This tiered approach ensures that agents get the best of all worlds: the deep relational knowledge from L3, the curated truth from L2, and the real-time operational state from L1.
