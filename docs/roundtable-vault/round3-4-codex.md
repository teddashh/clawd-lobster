## Unified Decision

### Dispute 1: Schema Rigidity vs Flexibility
**Decision: Claude wins: typed core tables with JSON overflow.**

Use fixed columns for fields that must be filtered, joined, ranked, audited, or indexed: lifecycle state, timestamps, entity IDs, aliases, edge types, confidence, provenance, chunk ordinals, etc.

Use `extra_json` only for non-contractual enrichment metadata. No EAV/property-table model. EAV hides complexity, weakens constraints, and makes high-signal queries harder to optimize.

### Dispute 2: Table Count / Complexity
**Decision: 7 Vault-owned tables.**

Codex’s 12-table design is too rigid for a new L3 memory layer. Gemini’s abstract graph is too loose. The final design uses a compact typed graph/evidence model with separate first-class aliases, chunks, and lifecycle log.

### Dispute 3: UUID vs IDENTITY PKs
**Decision: Claude wins for primary keys: `NUMBER GENERATED ... AS IDENTITY`.**

Use numeric identity primary keys for Oracle-local performance, smaller indexes, and simpler joins.

Add `public_id VARCHAR2(64)` as a unique external identifier on externally referenced tables. This gives portability/export stability without making every internal join pay the UUID cost.

## Final Table Set

1. `vault_records` — Canonical L3 memory records: facts, decisions, events, notes, summaries, artifacts, with typed fields plus `extra_json`.

2. `vault_chunks` — Chunked text/body slices for retrieval, embeddings, citation, and context assembly.

3. `vault_entities` — Canonical entities detected or declared in the vault: people, projects, systems, files, concepts, organizations.

4. `vault_entity_aliases` — First-class aliases, spellings, handles, paths, and alternate names mapped to canonical entities.

5. `vault_edges` — Typed graph relationships between records and/or entities, with confidence and provenance.

6. `vault_evidence` — Evidence links connecting records/edges/entities to supporting chunks, source spans, or imported memory layers.

7. `vault_lifecycle_events` — Immutable append-only lifecycle/audit log for creation, enrichment, merge, supersede, deprecate, delete-request, and restore events.

## Core Record Shape

Each major table uses:

```sql
id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
public_id VARCHAR2(64) UNIQUE NOT NULL
lifecycle_state VARCHAR2(32) NOT NULL
occurred_at TIMESTAMP WITH TIME ZONE NULL
ingested_at TIMESTAMP WITH TIME ZONE NOT NULL
created_at TIMESTAMP WITH TIME ZONE NOT NULL
updated_at TIMESTAMP WITH TIME ZONE NOT NULL
confidence NUMBER(5,4) NULL
extra_json CLOB CHECK (extra_json IS JSON)
```

`occurred_at` means when the remembered thing happened.

`ingested_at` means when Oracle learned it.

## Enrichment Pipeline

1. **Ingest** — Accept source payload through `vault_api`, assign `public_id`, record `ingested_at`, preserve raw source pointer.

2. **Normalize** — Clean text, extract metadata, classify record type, assign initial lifecycle state.

3. **Chunk** — Split long bodies into ordered chunks with stable chunk hashes and offsets.

4. **Embed** — Generate embeddings for chunks and optionally summaries, stored behind API-managed retrieval indexes.

5. **Extract Entities** — Detect entity mentions, resolve against aliases, create unresolved entities when needed.

6. **Alias Resolution** — Add or link aliases in `vault_entity_aliases`; never bury aliases only in JSON.

7. **Relation Extraction** — Create typed `vault_edges` between entities and records with confidence scores.

8. **Evidence Linking** — Attach each extracted entity, edge, or synthesized record to supporting chunks via `vault_evidence`.

9. **Lifecycle Append** — Write immutable `vault_lifecycle_events` for every material state transition or enrichment action.

10. **Publish for Query** — Mark records/chunks/edges as queryable only after minimum evidence and lifecycle checks pass.
